## Key Learnings

- **Project:** adoptu
- **Description:** A Kotlin pet adoption web application with **FIDO2/WebAuthn** passwordless authentication. Multi-page application with HTML rendered from Kotlin (kotlinx.html).
- **Gradle multi-module:** Root `build.gradle.kts` uses `apply false` plugins + explicit `tasks.register("test")` delegating to `:backend:test` AND `:frontend:jsTest`. The root project never applies Kotlin directly.
- **Kotlin/JS yarn tasks:** `kotlin.js.yarn=false` in `gradle.properties` does NOT work in Kotlin 2.3.20 — yarn still runs (`kotlinYarnSetup`, `kotlinNpmInstall`). `KotlinNpmInstallTask` extends `DefaultTask` (not `Exec`), so `.environment()` is unavailable. The only way to inject `NODE_OPTIONS` is in `gradlew` before the JVM starts.
- **Node.js 22 + yarn 1.x:** DEP0169 (`url.parse()`) and DEP0040 (`punycode`) warnings come from yarn 1.22.17 internals. Fixed by exporting `NODE_OPTIONS="--no-deprecation"` in `gradlew`.
- **Frontend test task name:** Kotlin/JS with `js(IR)` target uses `jsTest` (not `test`) — the root test task must `dependsOn(":backend:test", ":frontend:jsTest")`.

- **DatabaseFactory.listOfTables must be kept in sync with Models.kt:** Every `Table` object defined in `Models.kt` must be added to `listOfTables` in `DatabaseFactory.kt` — omitting one means `SchemaUtils.create()` never creates that table, causing a PSQLException at runtime. Tables previously missing: `UserSterilizationLocations`, `UserShelters`, `PasswordResetTokens`, `EmailChangeTokens`.

- **`npx sass` fails in this sandbox** (`npm ERR! Cannot read properties of undefined (reading 'stdin')`). Use the vendored dart-sass binary directly instead: `.gradle/sass/1.54.0/dart-sass/sass backend/src/main/scss/<name>.scss backend/src/main/resources/static/css/<name>.css --no-source-map`. Only top-level `.scss` files (not `_partial.scss` files) compile to their own `.css` — check which top-level files `@use`/import a changed partial (e.g. `grep -l "<partial-name>" backend/src/main/scss/*.scss`) and recompile each of those too.

- **kotlinx-serialization can't serialize a raw `Map<String, Any?>`/`call.respond(mapOf(...))` whose values mix multiple runtime types within one map.** Ktor's `KotlinxSerializationConverter` falls back to `guessSerializer()` (`io.ktor.serialization.kotlinx.SerializerLookup`) whenever a static type has no real serializer (e.g. generic `Any`/`Map<String, Any>`). That fallback requires every non-null value in the collection/map to resolve to the *same* `KSerializer` (`elementSerializer` does `distinctBy { it.descriptor.serialName }` and `error(...)`s if `size > 1`). A single-key map (`mapOf("blocked" to blocked)`) is fine; a map mixing e.g. Int + String + Long + Boolean across different keys throws `IllegalStateException` at serialize time, which Ktor's default pipeline turns into a 500 — even though the route code itself looks correct. Found this affecting `PhotographerService`'s request-map builders and `TemporalHomeRoutes`'s `/request` success payload (logged as bug-056). Fix is to respond with a proper `@Serializable` DTO instead of a raw mixed-type map, not to special-case Ktor.

- [2026-06-21] Do NOT add `kotlin.js.yarn=false` to `gradle.properties` — it has no effect in Kotlin 2.3.20 and yarn tasks still run. For Node.js 22 deprecation warnings from yarn, set `NODE_OPTIONS=--no-deprecation` in `gradlew` instead.
- [2026-06-21] Do NOT forget `:frontend:jsTest` when registering root `test` task — using only `:backend:test` silently skips all frontend tests.

- [2026-06-21] Do NOT define a new `Table` object in `Models.kt` without also adding it to `listOfTables` in `DatabaseFactory.kt`. Missing entries silently skip table creation and cause `PSQLException: relation "..." does not exist` at runtime.

- [2026-06-22] Do NOT edit CSS files directly. All styles live in `backend/src/main/scss/`. Shared/nav styles go in `_layout.scss`; page-specific styles in their own `.scss` file. After editing SCSS, recompile: `npx sass backend/src/main/scss/<name>.scss backend/src/main/resources/static/css/<name>.css --no-source-map`
- [2026-06-30] Do NOT add a `t('newKey')` call in a page's static JS (e.g. `shelters.js`) without also adding `"newKey" to "..."` for all 5 languages in `frontend/src/jsMain/kotlin/com/adoptu/frontend/I18n.kt`. The live translation system is the Kotlin/JS `I18n` object compiled into `common.js` and exposed as `window.AdoptuI18n`/`window.t` (wired in `Shared.kt`'s `commonScripts`) — it falls back to rendering the raw key string when a key is missing, which is invisible until someone hits that code path. The legacy `backend/src/main/resources/static/js/i18n/i18n.js` + `i18n-*.js` files are DEAD CODE — no page template loads them — so editing them has zero effect; always edit `I18n.kt`. After editing `I18n.kt`, rebuild with `./gradlew :frontend:jsBrowserProductionWebpack` to regenerate `common.js` (it auto-copies into `backend/src/main/resources/static/js/`). Recurring bug class — see bug-051.
- [2026-06-30] New git worktrees in this repo are missing `gradle/wrapper/gradle-wrapper.jar` (it's untracked by git, only `gradle-wrapper.properties` is). Before running `./gradlew` in a fresh worktree, `cp` the jar from the main checkout's `gradle/wrapper/gradle-wrapper.jar` into the worktree first, or `./gradlew` fails with "Unable to access jarfile".
- [2026-06-30] Merging a worktree branch that touches `frontend/src/jsMain/kotlin/**` into a `main` that has since regenerated its own `common.js`/`common.js.map` WILL conflict on those two generated bundle files even when the Kotlin source merges cleanly — they're build output, not source. Resolve with `git checkout --ours` on the bundle files (keep main's pre-merge bundle) and then rebuild via `./gradlew :frontend:jsBrowserProductionWebpack` post-merge to regenerate them from the merged source, rather than trying to hand-resolve the minified diff.
- [2026-06-30] Do NOT mix Base64 encoder/decoder variants in CryptoService. `encrypt()` uses `Base64.getUrlEncoder()` (produces URL-safe `_` and `-`). Both `decrypt()` and `decryptWithKey()` MUST use `Base64.getUrlDecoder()`. Using the standard `Base64.getDecoder()` silently returns null and causes all password operations to fail (15 tests).

- [2026-06-30] To E2E-test a Ktor route file that `inject<WebAuthnService>()`, the Koin test module must bind the FULL dependency chain, not just WebAuthnService itself: `Clock`, `UserRepositoryPort` (-> `UserRepository`), `NotificationPort` (-> `MockNotificationAdapter`), `UserService`, `EmailVerificationService(userRepo, notificationPort, clock)`, `PasswordService(userRepo, notificationPort, clock, baseUrl)`, `MagicLinkService(userRepo, notificationPort, clock, baseUrl, emailVerificationService)`, then `WebAuthnService(clock, emailVerificationService, userService, passwordService, magicLinkService, adminEmail, rpId, rpName, origins)` — all via `get()`, Koin infers type from constructor param position. Since `by inject<WebAuthnService>()` in `uiRoutes()`/similar is a lazy local delegate, routes that never touch the property (most page-render routes) won't trigger Koin resolution of this chain at all, but any route that does (e.g. `/verify`, `/verify-email`) will fail at request time with a Koin `NoBeanDefFoundException` if the chain is incomplete — copy the exact `single { WebAuthnService(...) }` block from `UsersRoutesE2ETest.kt` rather than re-deriving it.
- [2026-06-30] To assert a Ktor route's redirect status/Location header in a `testApplication` E2E test (rather than following the redirect to its 200 destination), create the test client with `createClient { followRedirects = false }` — the default `client` property follows redirects automatically (`HttpClientConfig.followRedirects = true` is the engine default), which silently turns a 302 assertion into a 200 assertion if not disabled. To carry a session cookie across requests in the same test, also `install(HttpCookies)` on that client — `testApplication`'s default `client` does not have cookie persistence by default. No existing helper route signs a session cookie directly; the simplest approach to get an authenticated session in a test is to add a test-only route (e.g. `get("/__test/login/{id}") { call.sessions.set(SessionUser(...)) }`) inside the test's own `routing {}` block alongside the real routes under test, then hit it once before the assertions — `configureSessions()`'s `SessionTransportTransformerMessageAuthentication` signs the `Set-Cookie` automatically on response.
- [2026-06-30] Do NOT assume `input`/`select`/`textarea` rules automatically pick up dark theme colors — the project sets `--bg`/`--text` as CSS vars but several rules (`form input/select/textarea`, `.auth-form input/select/textarea`, `.location-search-form` inputs) never set `background`/`color` explicitly, so they fell back to browser-default white. When adding/touching any input rule, always set `background: var(--bg); color: var(--text);` explicitly, and add the `-webkit-autofill` override (see `_base.scss`) since Chrome/Edge force a white autofill background unless overridden.

- [2026-06-30] **Kover setup:** `kotlinx.kover` plugin 0.9.1 fails on Kotlin 2.4.0 (`Could not get unknown property 'compileKotlinTask'`); use 0.9.8+. Kover auto-instruments ALL `Test`-type tasks in a project (including manually `tasks.register<Test>(...)` ones), so `:backend:test` + `:backend:integrationTest` coverage merges automatically into one report — no extra wiring needed beyond applying the plugin.
- [2026-06-30] **`integrationTest` task was completely broken (3 compounding bugs), fixed:** (1) `tasks.withType<Test> { exclude("**/*IT.class") ... } }` applies to every Test task including ones registered later, so `integrationTest`'s own `filter { includeTestsMatching(...) }` had nothing left to match — needed `setExcludes(emptySet())` inside the `integrationTest` registration. (2) The task never had `testClassesDirs`/`classpath` set (only JVM Test Suite-managed tasks get that automatically) — needed `testClassesDirs = sourceSets["test"].output.classesDirs` and `classpath = sourceSets["test"].runtimeClasspath`. (3) `includeTestsMatching(".*IT")` used regex syntax but Gradle's TestFilter is `*`-glob only (`.` is literal) — needed `"*IT"` not `".*IT"`. All three together meant `./gradlew :backend:integrationTest` had **always** reported BUILD SUCCESSFUL while running zero tests, for the entire project history. Once fixed, DatabaseFactoryInitIT/ImageStorageIT/SterilizationLocationRepositoryIT/ApplicationTestcontainersIT all pass for real against Testcontainers Postgres/LocalStack.
- [2026-06-30] **Global Koin context (`startKoin{}`/`stopKoin()`) is unsafe across test classes sharing a JVM fork.** If one test class calls raw `org.koin.core.context.startKoin{}` (e.g. for a `KoinComponent`/`by inject()` service under test) without a prior `stopKoin()`, it throws `KoinApplicationAlreadyStartedException` whenever another test class in the same forked JVM (e.g. a Ktor `testApplication` that does `install(Koin){...}`) left a global Koin instance running. Fix: always call `stopKoin()` (safe no-op if nothing is running — `GlobalContext.stopKoin()` is `_koin?.close(); _koin = null`) immediately before `startKoin{}` in `@BeforeEach`.
- [2026-06-30] **`configureSerialization()` sets `prettyPrint = true`** on the JSON content negotiation — response bodies have spaces after colons (`"userId": 1`, not `"userId":1`). Any test asserting `body.contains("\"key\":value")` without the space will fail. Check this project's Serialization.kt before writing string-contains JSON assertions; prefer decoding the body with `Json.decodeFromString` over substring matching where practical.
- [2026-06-30] **kotlinx.html: setting an attribute (e.g. `onClick = "..."`) AFTER adding child content (e.g. `+"text"`) inside the same tag builder throws `IllegalStateException: You can't change tag attribute because it was already passed to the downstream` at render time** — found in `UIRoutes.kt`'s block-rescuer page (bug-057), a real production 500 that predated any test work. Attributes must be set before children in every `kotlinx.html` tag block.
- [2026-06-30] **`respondData(result: ServiceResult<T>)` unwraps `Success` to `result.data` directly (`respond(result.data as Any)`)** — for `ServiceResult<Unit>` (e.g. delete endpoints) this serializes to `{}`, NOT `{"success":true}`. Only `respondSuccess()` wraps in `SuccessResponse(success=true)`. Check which responder a route handler actually calls before asserting response body shape.
- [2026-06-30] Several `*RepositoryPort`/`UserRepository.kt` methods (`updatePhotographerSettings`, parts of `getPhotographers`) are dead/duplicate code — the real production photographer-settings route goes through `PhotographerService` → `PhotographerRepositoryImpl`, not `UserService` → `UserRepository`, even though `UserRepository` has a full parallel implementation satisfying the same interface method. Also `DataResponder`/`SuccessResponder`/`CustomResponder` in `plugins/Responses.kt` and `magicLinkLoginPage()`/`guestNav()`/`DIV.languageDropdown()` in `pages/` are unused — candidates for cleanup, excluded from Kover coverage accounting (DataResponder/SuccessResponder/CustomResponder) rather than fake-tested.
