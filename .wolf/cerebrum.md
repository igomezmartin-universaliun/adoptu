## Key Learnings

- **Project:** adoptu
- **Description:** A Kotlin pet adoption web application with **FIDO2/WebAuthn** passwordless authentication. Multi-page application with HTML rendered from Kotlin (kotlinx.html).
- **Gradle multi-module:** Root `build.gradle.kts` uses `apply false` plugins + explicit `tasks.register("test")` delegating to `:backend:test` AND `:frontend:jsTest`. The root project never applies Kotlin directly.
- **Kotlin/JS yarn tasks:** `kotlin.js.yarn=false` in `gradle.properties` does NOT work in Kotlin 2.3.20 â€” yarn still runs (`kotlinYarnSetup`, `kotlinNpmInstall`). `KotlinNpmInstallTask` extends `DefaultTask` (not `Exec`), so `.environment()` is unavailable. The only way to inject `NODE_OPTIONS` is in `gradlew` before the JVM starts.
- **Node.js 22 + yarn 1.x:** DEP0169 (`url.parse()`) and DEP0040 (`punycode`) warnings come from yarn 1.22.17 internals. Fixed by exporting `NODE_OPTIONS="--no-deprecation"` in `gradlew`.
- **Frontend test task name:** Kotlin/JS with `js(IR)` target uses `jsTest` (not `test`) â€” the root test task must `dependsOn(":backend:test", ":frontend:jsTest")`.

- **i18n strings are duplicated in two places:** Each page's Kotlin (e.g. `SheltersPage.kt`) renders a hardcoded English fallback string with `attributes["data-i18n"] = "<key>"`, while `frontend/src/jsMain/kotlin/com/adoptu/frontend/I18n.kt` holds the same key translated across 5 locales (en, es, fr, pt, zh). Both must be updated together when correcting copy â€” editing only one leaves a stale fallback or stale translation.

- **DatabaseFactory.listOfTables must be kept in sync with Models.kt:** Every `Table` object defined in `Models.kt` must be added to `listOfTables` in `DatabaseFactory.kt` â€” omitting one means `SchemaUtils.create()` never creates that table, causing a PSQLException at runtime. Tables previously missing: `UserSterilizationLocations`, `UserShelters`, `PasswordResetTokens`, `EmailChangeTokens`.

- **`npx sass` fails in this sandbox** (`npm ERR! Cannot read properties of undefined (reading 'stdin')`). Use the vendored dart-sass binary directly instead: `.gradle/sass/1.54.0/dart-sass/sass backend/src/main/scss/<name>.scss backend/src/main/resources/static/css/<name>.css --no-source-map`. Only top-level `.scss` files (not `_partial.scss` files) compile to their own `.css` â€” check which top-level files `@use`/import a changed partial (e.g. `grep -l "<partial-name>" backend/src/main/scss/*.scss`) and recompile each of those too. (If the vendored binary is unavailable, see the `npm pack`-based fallback in Do-Not-Repeat below.)
- **`.pet-card` markup is duplicated across pages, not shared via a component:** `backend/src/main/resources/static/js/index.js` (home/browse page) and `my-pets.js` (My Pets page) both hand-build the same `.pet-card` HTML string against the shared `.pet-card`/`.pet-info`/`.pet-name` rules in `style.scss`. `index.js` is the more complete/correct reference implementation (it wraps age/rescue-date in `.label`/`.value` spans as the CSS expects). When fixing pet-card display bugs, check both files for the same divergence â€” a fix applied to only one page will leave the other inconsistent.
- **`Country` (backend/src/main/kotlin/com/adoptu/common/Country.kt) is the single source of truth for country values.** The dropdown (`Shared.kt: countrySelect()`), DB storage (`Models.kt` columns use `enumerationByName("country", 100, Country::class)`), and validation (every repository touching a `country` column calls `Country.fromDisplayName()` before reading/writing) all derive from this enum. DTOs stay `String` on the wire (`country: String`) â€” repositories parse at the boundary, throwing `IllegalArgumentException` on insert/update with an unrecognized value, and returning `emptyList()` on a search filter with an unrecognized value. `Country.fromDisplayName()` does accent/case-insensitive fallback matching, so stray input (e.g. `"MĂ©xico"` vs `"Mexico"`) still resolves instead of silently returning zero results.
- **Test fixtures must use real canonical country names from `Country.kt`, not placeholders.** Several test files used `"USA"` as a stand-in country value â€” it was never a real dropdown option (the canonical value is `"United States"`) and broke once country columns became enum-backed. Always use an exact `Country.entries` `displayName` in test data.
- **There are TWO parallel i18n systems and only one is live:** `frontend/src/jsMain/kotlin/com/adoptu/frontend/I18n.kt` (Kotlin, compiled into `common.js` via `:frontend:jsBrowserProductionWebpack`, exposed as `window.AdoptuI18n`) is what every page actually uses â€” `Shared.kt`'s `commonScripts()` wires `window.t` to it. The static files under `backend/src/main/resources/static/js/i18n/i18n-*.js` (plain JS, with their own `t()`/`i18n.translations`) are NOT loaded by any page (`Shared.kt` never adds a `<script>` tag for them) â€” they're dead code. Always add new translation keys to `I18n.kt`, never to `static/js/i18n/i18n-*.js`, and grep `t('keyName')` against `I18n.kt` (not the static i18n files) when chasing an "untranslated key shown literally" report.
- **After editing I18n.kt, rebuild the bundle:** `./gradlew :frontend:jsBrowserProductionWebpack` (task `:frontend:copyJsToBackendResources` then copies the output into `backend/src/main/resources/static/js/common.js`/`.js.map`). Editing the Kotlin source alone has zero effect until this runs.
- **`gradle/wrapper/gradle-wrapper.jar` is now tracked by git** (fixed 2026-06-30 â€” a `.gitignore` rule-order bug previously shadowed the negation that was supposed to un-ignore it; see Do-Not-Repeat). If `./gradlew` ever fails again with "Unable to access jarfile" in a fresh checkout, check `.gitignore` rule order before assuming it needs a manual copy.
- **EnterWorktree's default `fresh` baseRef branches from `origin/main`, which can be far behind local `main`** when prior sessions merged work locally without pushing. Before editing files that recent sessions touched (e.g. `I18n.kt`), diff the worktree branch against local main's HEAD; if stale, `git merge <local-main-sha>` into the worktree branch before starting work (or after, before merging back) to avoid silently reverting recent fixes.

Don't trust buildspec.yml/README deploy docs as ground truth for what's actually running â€” verify against the live account.
- **Live prod had real bugs worth knowing about**: ECS task definition hardcoded `ADOPTU_DB_PASSWORD` in plaintext (same value as the documented dev default in AGENTS.md); the IAM task policy's S3 statement was scoped to a literal unfilled placeholder ARN (`your-bucket-name`); the container port mapping (80) didn't match what the app actually listens on (8080, per Dockerfile/application.conf). - **adopt-u's CloudFront app origin is `http-only` on port 8080 (no TLS to origin) by design, verified against the live distribution's actual `CustomOriginConfig`** â€” not a security gap to "fix" by adding a load balancer. Without an LB there's no AWS-managed TLS termination point in front of the app; encrypting that hop would require the Ktor container terminating TLS itself (Netty/Ktor SSL config + ACM Private CA, since plain ACM certs can't be installed on self-managed compute), which hasn't been requested. When asked to write IaC "matching the live deployment," check the actual running resource's config via the API before assuming a setting looks wrong (e.g. `aws cloudfront get-distribution`) rather than improving it unprompted.
- **RDS in a VPC with IPv6-only ECS subnets needs `network_type = "DUAL"`**, not just IPv4 â€” an IPv6-only Fargate task has no IPv4 address on its ENI at all, so it cannot open an IPv4 TCP connection to an IPv4-only RDS endpoint even though they're in the same VPC. The DB subnet group must support `DUAL` (check `aws rds describe-db-subnet-groups`) before this is possible; intra-VPC routing then handles it with no NAT/IGW involved either way.
- **For "no NAT" + IPv6-only ECS Fargate**: a VPC with an Amazon-provided IPv6 `/56` (the default for newer default VPCs) is "public" IPv6 â€” the VPC's main route table already has `::/0 -> igw`, so IPv6-only subnets get direct internet egress through the regular Internet Gateway with zero extra routing config (no NAT Gateway, no Egress-Only IGW needed). ALBs, however, still require a subnet with an IPv4 CIDR present (even when using `ip_address_type = "dualstack-without-public-ipv4"` to avoid consuming a public IPv4 address) â€” pure IPv6-only subnets work for ECS tasks but not for ALB nodes; put the ALB in the existing dual-stack subnets and only the ECS tasks in the IPv6-only ones.

- [2026-06-30] When introducing a new internal-only DNS hostname (e.g. for a CDN origin), it CANNOT reuse a name that's already a public alias to something else (e.g. `api.adopt-u.org` already aliases to the CloudFront distribution itself) - a single DNS name can only be one record type. Give the internal-only hostname its own distinct name (adopt-u uses `backend.adopt-u.org`); don't assume a semantically-fitting public name is free to reuse for internal plumbing without checking what it already points to.
- **The root Dockerfile had never actually built successfully** (confirmed 2026-06-30 by running `docker build` directly): it referenced a nonexistent `:backend:compileSass` Gradle task AND a nonexistent `:backend:shadowJar` task (Shadow plugin was never applied). Don't assume the Dockerfile works just because it's been merged/unchanged for a while â€” verify with an actual `docker build` before relying on or further editing it. See bug-067/068.
- **SCSSâ†’CSS compilation is a manual dev-time step, not part of any build.** There is no Gradle Sass task and never was. Compiled `.css` files live committed in `backend/src/main/resources/static/css/` (see the existing `npx sass`/vendored-dart-sass workflow notes below) and are served from the jar's classpath via Ktor's `staticResources("/static", "static")`. The Dockerfile must not try to invoke a Sass build step.
- **backend uses `id("com.gradleup.shadow") version "9.4.3"`** (the maintained fork of the old `com.github.johnrengelman.shadow`, which is dead) to produce `backend/build/libs/*-all.jar`, the fat jar the Dockerfile packages. Keep the Shadow plugin version aligned with Gradle's major version (9.x plugin for Gradle 9.x).
- **`backend/build.gradle.kts`'s `kotlin { jvmToolchain(N) }` must match the JDK actually present in whatever image runs the build** (currently 25, matching `amazoncorretto:25-alpine-jdk` in the Dockerfile's builder stage). Toolchain auto-provisioning/download is not configured, so a mismatch fails the build immediately rather than degrading.
- **jlink-ing a custom runtime JRE: the JDK used as the `--module-path` source for `jlink` and the OS base image of the final runtime stage must share the same libc.** A JDK built against glibc (e.g. plain `amazoncorretto:25`) jlinked and then copied into a musl-based image (e.g. `alpine:*`) produces a `java` binary that fails to exec at all (`sh: java: not found` â€” a misleading error; it's actually a dynamic-linker/ABI failure, not a missing-file issue). Use the libc-matching JDK image (e.g. `amazoncorretto:25-alpine-jdk`) as the jlink source when the runtime stage is Alpine, and pin both stages to the *same* Alpine minor version to avoid musl ABI drift between versions. See bug-071.

- **kotlinx-serialization can't serialize a raw `Map<String, Any?>`/`call.respond(mapOf(...))` whose values mix multiple runtime types within one map.** Ktor's `KotlinxSerializationConverter` falls back to `guessSerializer()` (`io.ktor.serialization.kotlinx.SerializerLookup`) whenever a static type has no real serializer (e.g. generic `Any`/`Map<String, Any>`). That fallback requires every non-null value in the collection/map to resolve to the *same* `KSerializer` (`elementSerializer` does `distinctBy { it.descriptor.serialName }` and `error(...)`s if `size > 1`). A single-key map (`mapOf("blocked" to blocked)`) is fine; a map mixing e.g. Int + String + Long + Boolean across different keys throws `IllegalStateException` at serialize time, which Ktor's default pipeline turns into a 500 â€” even though the route code itself looks correct. Found this affecting `PhotographerService`'s request-map builders and `TemporalHomeRoutes`'s `/request` success payload (logged as bug-056). Fix is to respond with a proper `@Serializable` DTO instead of a raw mixed-type map, not to special-case Ktor.

- [2026-06-30] **Adding a required query param to a shared "list" endpoint can silently break internal consumers of that same endpoint.** `GET /api/pets` is used both by the public browse page (`index.js`) and by the rescuer/admin management page (`my-pets.js`, via `pets.filter(p => p.rescuerId === user.id)`). Making `country` mandatory + exact-match-filtering there (per the shelters/sterilization-locations pattern) would have hidden legacy pets with no country set from their own owner's management page. Fixed by adding a separate unfiltered `GET /api/pets/mine` (session-scoped, any status/country) for the management view, and keeping `GET /api/pets` strictly country-required for public search. Before making an existing repository/service method's parameter required, grep every route AND every static JS file calling that endpoint/method, not just the obvious one from the feature brief.
- [2026-06-30] `tokensave` MCP's index can point at an entirely different repo/worktree than the one you're actually working in (`tokensave_status` reports `worktree_mismatch` with the wrong `index_root`) â€” always check `tokensave_status` first; if it mismatches, fall back to plain Read/Grep instead of trusting search results from the wrong codebase.
- [2026-06-30] Ktor route trees resolve literal path segments before parameter segments at the same depth regardless of declaration order â€” confirmed by adding `get("/mine")` alongside an existing `get("/{id}")` under `/api/pets` (matching the pre-existing `get("/my-adoption-requests")` sibling). No need to register literal routes before parameterized ones.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->
- [2026-06-30] Runtime image switched from `amazoncorretto:25-alpine-jdk` (full JDK, ~190MB+ layer) to a custom `jlink`-built minimal JRE on plain `alpine`, derived per-build via `jdeps --print-module-deps` against the actual fat jar plus a small hardcoded safety-net module list (`jdk.crypto.ec`, `jdk.naming.dns`, `jdk.unsupported`, `java.naming` â€” TLS/DNS/Unsafe-reflection paths static `jdeps` analysis is known to miss for Netty/AWS-SDK-style libraries). Considered hand-picking a fixed module list instead of deriving via `jdeps`, rejected it as too fragile to dependency changes (silent breakage on a future library bump). Verified end-to-end by actually booting the built image against a real local Postgres container and confirming `Application started` + HTTP 200, not just a successful `docker build` â€” a build that completes doesn't prove a jlinked runtime can actually exec (see bug-071). Net result: ~124MB final image vs. the prior full-JDK Alpine base, faster ECR pull/cold start on Fargate.
- [2026-06-30] Declined to bake AppCDS (`-XX:ArchiveClassesAtExit`/`-XX:+AutoCreateSharedArchive`) into the image. Tried `-XX:+AutoCreateSharedArchive` as a runtime-only JAVA_OPTS flag; it only logged a CDS warning on every boot with zero benefit, because ECS Fargate tasks get a fresh ephemeral filesystem per launch â€” the archive file never survives to be reused on a second run. A real build-time-baked AppCDS archive would need a representative training run against live DB/AWS endpoints, which isn't safely scriptable inside the Dockerfile; skipped as not worth the fragility for this deployment model. Revisit only if ECS moves to long-lived/restart-in-place tasks.
- [2026-06-30] Country values switched from free-text `varchar` columns to an Exposed `enumerationByName` column backed by a new `Country` enum, after a Mexico search returned zero results due to an accent mismatch (`"Mexico"` dropdown value vs `"MĂ©xico"` stored in 2 shelter rows). Considered: (1) normalize the bad DB rows only, (2) make the filter comparison accent-insensitive, (3) make `Country` an enum and the single source of truth end-to-end. Chose (3), per explicit user direction, to eliminate the whole class of dropdown/DB drift rather than patch one symptom â€” affects 6 tables (shelters, sterilization locations, user shelters, user sterilization locations, temporal homes, photographers) that all shared the same free-text-country + exact-match-filter pattern.

- [2026-06-30] After fixing a serialized-DB-call bug by wrapping repositories in `withContext(Dispatchers.IO) { transaction { ... } }` (commit 95a5ce8), load-testing against the *actual* container resource cap (`--cpus=0.5 --memory=1024m`, matching ECS Fargate) showed this regressed throughput from 18.2 RPS to 0.1 RPS. `Dispatchers.IO` is elastic/unbounded (up to 64 threads) and assumes I/O-blocked threads don't cost real CPU â€” under a cgroup CPU cap that assumption is false, so 70-84 threads just thrash for the same half-core instead of adding parallelism. Fixed with a single shared bounded dispatcher (`Dispatchers.IO.limitedParallelism(4)`, `backend/src/main/kotlin/com/adoptu/adapters/db/DbDispatcher.kt`) reused across all 8 repositories. Lesson: an unbounded elastic dispatcher is not automatically "safe" just because it's the idiomatic choice for blocking I/O â€” always size dispatcher parallelism relative to the container's actual CPU allocation, and load-test fixes against real resource limits (not just correctness) before trusting them under production constraints.
- [2026-06-30] Building Terraform/OpenTofu for an already-live AWS account (adopt-u prod): chose a hybrid brownfield strategy rather than either pure greenfield (would fail immediately - S3 bucket names, the RDS identifier, and CloudFront aliases are already taken in the account) or pure in-place mutation of the live single-replica ECS service (no redundancy, real risk of an outage mid-import). Stateful/identity-bound resources (S3 buckets, RDS instance, all 3 CloudFront distributions, the existing IAM role/policy names, most Route53 records) are written to match live config exactly and shipped with `tofu import` commands. Compute (ECS cluster/service/task def/security groups) is stood up as brand-new, freely-named resources that don't collide with anything, deployed in parallel alongside the still-running old service; the README documents a 4-step cutover (import -> stand up new compute -> manually re-point the `ecs.<domain>` record at the new task -> decommission the old hand-made resources) rather than one big-bang apply.
- [2026-06-30] First draft of this stack added an ALB + target group in front of the ECS service "to fix" the live deployment's fragile hardcoded-task-IP CloudFront origin. User rejected this forcefully and explicitly stated it had already been said before ("AGAIN!!"). Reverted: no load balancer of any kind. The fragility (DNS goes stale when the task restarts/redeploys) is solved instead by documenting a manual re-point step in the README, not by introducing new infrastructure the user didn't ask for. General lesson: don't "fix" something the user didn't flag as broken just because it looks fragile from the outside - ask, or at minimum match the live config exactly and call out the tradeoff in prose rather than silently architecting around it.
- [2026-07-01] `kotlin-js-store/` was gitignored, meaning `yarn.lock` (managed by the Kotlin/JS Gradle plugin) was never committed and could drift between machines, causing `Execution failed for task ':kotlinStoreYarnLock' ... If this error recurs after a legitimate dependency bump, rerun `kotlinUpgradeYarnLock` and commit the updated lock file rather than deleting/re-ignoring it. See bug-079.
